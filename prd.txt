1. Feature Overview
This PRD covers two distinct but related Google integrations for form. They are separate in implementation but designed to work together seamlessly from the user's perspective.

Part A — Authentication (Google Sign-In & Email/Password)

2. Authentication
2.1 Overview
form. supports two authentication methods: Google Sign-In for users with a Google account, and Email/Password for users who prefer not to use Google. Both are handled entirely by Supabase Auth. No third-party auth SDK is needed beyond what Supabase provides.

Scope decision
Both methods create the same type of account in Supabase. A user who signs up with email can later link their Google account if they choose, but this is a v1.1 consideration. In v1.0 the two methods are independent.

2.3 Google Sign-In Flow
	•	User taps 'Continue with Google'
	•	Native Google account picker opens
	•	User selects their Google account
	•	Supabase checks if this Google account already exists in auth.users
	•	New user — Supabase creates auth.users record, triggers profiles insert, sets onboarding_complete = false — lands on onboarding
	•	Returning user — Supabase retrieves existing session, onboarding_complete = true — lands on Home dashboard

2.4 Email Sign-Up Flow
	•	User taps 'No account yet? Create one'
	•	Sign Up screen appears with three fields: email, password, confirm password
	•	Password requirements: minimum 8 characters — validated client-side before submission
	•	User taps 'Create account'
	•	Supabase sends a verification email to the provided address
	•	User sees a confirmation screen: 'Check your email — tap the link to verify your account'
	•	After clicking the verification link, user is redirected back to the app
	•	Supabase creates auth.users record, triggers profiles insert, sets onboarding_complete = false
	•	User lands on onboarding screen 1

Email verification is required
Users cannot access the app until they verify their email. If they try to sign in before verifying, show: 'Please verify your email first. Check your inbox or resend the link.'

2.5 Email Sign-In Flow
	•	Returning user enters email and password on the Sign In screen
	•	Taps 'Sign in'
	•	Supabase authenticates credentials
	•	On success — session created, user lands on Home dashboard
	•	On failure — show inline error: 'Incorrect email or password'

2.6 Forgot Password Flow
	•	User taps 'Forgot password?' on the Sign In screen
	•	A single field screen appears asking for their email address
	•	User enters email and taps 'Send reset link'
	•	Supabase sends a password reset email
	•	User sees: 'Reset link sent. Check your inbox.'
	•	After clicking the link, user is taken to a Reset Password screen with new password and confirm password fields
	•	On success — password updated, user is signed in and lands on Home dashboard

Returning user logic
If a returning user signs in with either method and their account exists in Supabase, they land directly on the Home dashboard with all data intact. No re-onboarding, no data loss.

2.8 Google Cloud Console Setup
Required steps before the integration works in production:
	•	Create a project in Google Cloud Console
	•	Enable the Google Identity API
	•	Create OAuth 2.0 credentials — application type: Web (for Supabase callback) and Android / iOS (for mobile app)
	•	Add authorised redirect URI: your Supabase project callback URL
	•	Add your app's bundle ID or package name to the OAuth client
	•	Copy Client ID and Client Secret into Supabase Auth settings

Cost
Google OAuth is completely free. Google Cloud Console account is free. No billing required for authentication at any scale.

	•	Settings screen includes a 'Sign out' option
	•	Calls Supabase signOut() which clears the local session
	•	User is returned to the Sign-In screen

2.11 Account Deletion

	•	Clears local session and returns user to Sign-In screen
	•	This action is irreversible — the confirmation dialog must state this explicitly



Part B — Google Fit Sync

3. Google Fit Weight Sync
3.1 Overview
Google Fit Sync imports the user's historical body weight data from Google Fit into form. This is a pull-only integration — form. reads weight data from Google Fit and stores it locally in Supabase. form. does not write back to Google Fit.

Scope — weight only
This integration syncs body weight data only. Steps, activity, heart rate, sleep, and other Google Fit data types are out of scope for v1.0.

3.2 User Flow
	•	After completing onboarding, user sees a prompt: 'Import your weight history from Google Fit?'
	•	User taps 'Connect Google Fit'
	•	Google OAuth consent screen appears, requesting fitness.body.read permission
	•	User grants permission
	•	App fetches all available weight entries from Google Fit (up to 12 months back)
	•	App shows a preview: 'Found 47 weight entries between Jan 2024 and Mar 2025'
	•	User taps 'Import' to confirm
	•	Entries are written to Supabase, duplicates skipped
	•	User sees import summary and lands on Home dashboard with data populated

This prompt is optional
If the user dismisses the Google Fit prompt, they are taken directly to the Home dashboard. They can trigger the sync later from Settings > Import data > Google Fit.

3.3 OAuth Scope Required
Scope
https://www.googleapis.com/auth/fitness.body.read
What it grants
Read-only access to body weight and related body measurements from Google Fit
What it does NOT grant
Access to activity, location, steps, heart rate, sleep, or any other data type
User sees
Google consent screen stating: 'form. wants to view your body measurements'

Separate OAuth token
The Google Fit OAuth token is separate from the Google Sign-In token. Sign-In uses basic profile scope. Google Fit requires the fitness.body.read scope. The user will see two separate permission prompts across the app lifecycle — one for sign-in, one for Google Fit.

3.4 Google Fit API — Data Fetching Logic
Use the Google Fit REST API to fetch weight data. The key endpoint is the Datasets endpoint under the Users.dataSources resource.

API

3.6 Duplicate Handling
	•	Before inserting each entry, check if a log_entries row already exists for that user on that date
	•	If a row exists — skip the import for that date, do not overwrite
	•	If no row exists — insert the new entry
	•	Count skipped entries and include in the import summary shown to the user

3.7 Import Summary Screen
After the sync completes, show a summary card before navigating to the Home dashboard:
	•	Total entries found in Google Fit
	•	Entries successfully imported
	•	Entries skipped (already existed in form.)
	•	Date range of imported data — e.g. 'Jan 2024 to Mar 2025'
	•	An undo option — 'Remove imported entries' — available for 60 seconds after import

3.8 Re-sync Behaviour
	•	User can trigger a manual re-sync at any time from Settings > Import data > Sync Google Fit
	•	Re-sync fetches entries from the last synced date to today — not a full 12-month re-fetch
	•	Store last_synced_at timestamp in the profiles table to track this
	•	No automatic background sync in v1.0 — manual only

3.9 Disconnect Google Fit
	•	User can revoke Google Fit access from Settings > Import data > Disconnect Google Fit
	•	This revokes the OAuth token stored in the app
	•	Existing imported data is NOT deleted — the user keeps their history
	•	User can reconnect at any time and re-sync

Knowing whether an entry came from manual logging, Google Fit, or CSV import lets the app display it differently in the history list (e.g. a small 'Google Fit' badge) and helps with duplicate detection logic during re-syncs.

6. Google Cloud Console Setup Checklist
Complete these steps before testing either integration on a real device.

6.1 For Google Sign-In
	•	Create a new project in Google Cloud Console
	•	Enable the Google Identity API
	•	Create OAuth 2.0 credentials (Web application type)
	•	Add Supabase callback URL to authorised redirect URIs
	•	Create separate OAuth credentials for Android and iOS with correct bundle ID / package name
	•	Add Client ID and Client Secret to Supabase Dashboard > Authentication > Providers > Google

6.2 For Google Fit
	•	In the same Google Cloud project, enable the Fitness API
	•	Add fitness.body.read to the OAuth consent screen scopes
	•	Complete the OAuth consent screen — app name, support email, privacy policy URL
	•	If app is not yet published, add test user Google accounts in the OAuth consent screen
	•	Note: Google Fit API does NOT require paid API access — it is free within standard quotas

App verification
For production release, Google requires OAuth app verification if you request sensitive scopes. fitness.body.read is classified as a sensitive scope. Submit for verification before App Store / Play Store launch. Review typically takes 1–3 weeks.


7. Constraints & Known Limitations
	•	Google Fit is Android-first. On iOS, Google Fit data availability depends on whether the user has the Google Fit app installed and syncing. Apple Health is the preferred data source for iOS users — out of scope for v1.0.
	•	Google Fit API quota is 10,000 requests per day per project — more than sufficient for v1.0 user volumes.
	•	The Google Fit app is being deprecated by Google in favour of Google Health Connect (Android 14+). This integration targets the current Fitness API which remains active. A Health Connect migration may be needed in v2.0.
	•	Google OAuth app verification is required before non-test users can use Google Fit sync in production. Plan 2–3 weeks for review before launch.



Part C — Two-Way Google Fit Sync (v1.1)

7. Two-Way Google Fit Sync (v1.1)
7.1 Overview
In v1.0, Google Fit sync is read-only — form. imports weight from Google Fit but does not write back. In v1.1, users can optionally enable two-way sync so that every weight entry logged in form. is also pushed to Google Fit automatically.

Opt-in only
Two-way sync is off by default. Users enable it in Settings > Google Fit > Sync weight back to Google Fit. Users who do not care about cross-app sync are not affected.

7.2 Additional OAuth Scope Required
New scope
https://www.googleapis.com/auth/fitness.body.write
Combined scopes
fitness.body.read + fitness.body.write
User sees
Updated consent screen: 'form. wants to view and add body measurements'
When requested
Only when user enables the two-way sync toggle in Settings — not during initial Google Fit setup

Re-consent required
Adding fitness.body.write requires the user to re-authorise Google Fit with the updated scope. When the user enables the toggle, trigger the OAuth flow again to request the write permission. Existing read token is insufficient.

7.3 Write Flow
	•	User logs a weight entry in form. (quick log or full entry screen)
	•	Entry is saved to Supabase as normal
	•	If two-way sync is enabled, form. immediately sends the entry to Google Fit via the API
	•	Google Fit stores the entry and it appears in the Google Fit app and any connected apps
	•	The write call is made asynchronously — it does not block the UI or delay the log confirmation

7.4 Google Fit API — Write Endpoint
API
Google Fitness REST API v1
Endpoint
users.dataSources.datasets.patch
Data type name
com.google.weight
Data source ID
raw:com.google.weight:your_app_package_name
Value to send
Weight in kilograms (float), with startTimeNanos and endTimeNanos set to the same timestamp

7.5 Failure Handling
The write to Google Fit is best-effort. A failure must never prevent the entry from being saved in form.
	•	If the Google Fit write call fails — save the entry to Supabase anyway, queue the Google Fit write for retry
	•	Retry up to 3 times with exponential backoff
	•	If all retries fail — entry remains in form. correctly, show a silent indicator in the log history (small icon) that this entry was not synced to Google Fit
	•	User can manually retry failed syncs from Settings > Google Fit > Retry failed entries

7.6 Retroactive Sync
	•	When a user first enables two-way sync, offer an option: 'Push all existing form. entries to Google Fit?'
	•	This is optional — user can choose to sync from today only
	•	If accepted, all existing log_entries for that user are sent to Google Fit in a single batch
	•	Duplicates in Google Fit (entries that already exist from the v1.0 import) are overwritten with the same value — no harm done

7.7 Supabase Schema Addition
New column
google_fit_write_enabled — boolean, default false, on the profiles table
New column
google_fit_write_token — text, nullable — stores the upgraded OAuth token with write scope
New column on log_entries
google_fit_synced — boolean, default false — tracks whether this entry has been pushed to Google Fit


8. Google Fit Data Source Strategy
8.1 The Problem with the Merged Feed
Google Fit does not decide which data is correct. It is a passive data store. When multiple apps write weight data to Google Fit, each writes under its own data source ID. The merged feed simply combines them — last write wins for the same timestamp. form. importing from the merged feed risks pulling in data from Withings scales, Garmin watches, MyFitnessPal, and other apps the user may not even recognise.

Core principle
form. should by default only import data that it can vouch for. The safest approach is to read from form.'s own data source first, and make mixed-source import an explicit opt-in choice.

8.2 Data Source Approach
Default import
Read only from form.'s own Google Fit data source (raw:com.google.weight:form.package.name). This contains only entries that form. itself previously wrote to Google Fit.
Mixed source import (opt-in)
Read from the merged feed (derived:com.google.weight:merge). Contains data from all connected apps. User must explicitly enable this in Settings > Google Fit > Include data from other apps.
Data source creation
form.'s own data source must be created via the API on first write. On first two-way sync enable, form. calls users.dataSources.create before any write.
Source label in history
Each imported entry shows a small source badge in the log history: 'Google Fit' for merged, 'form. backup' for own source.

8.3 Mixed Source Import — User Warning
When the user enables 'Include data from other apps', show a one-time warning:
Warning shown to user
'This will import weight entries from all apps connected to Google Fit — including smart scales, other fitness apps, and manual entries. Some entries may be duplicates or from devices you no longer use. You can review and delete them after import.'

8.4 Unit Handling
	•	Google Fit stores weight in kilograms internally regardless of what the user sees in the Google Fit app
	•	All values from the Google Fit API are already in kg — no conversion needed on import
	•	Round to 1 decimal place on import — Google Fit stores as float with many decimal places
	•	Validate range on import: discard any entry below 20 kg or above 300 kg as corrupted data

10. Open Questions
	•	Should the Google Fit sync prompt appear immediately after onboarding, or only after the user has manually logged at least one entry?
	•	Should deletion sync (removing entries from Google Fit when deleted in form.) be added in v1.2?
	•	Should we support Google Health Connect (Android 14+) in v1.0 as an alternative to the legacy Fitness API, given Google is pushing migration?
	•	For the Google Fit token — should it be stored in Supabase (current spec) or in the device keychain only? Keychain is more secure but limits server-side sync in future.
	•	For the mixed-source import toggle — should it be shown during the initial Google Fit setup flow, or only discoverable in Settings?
